Small Footprint Cim Broker

sfcb-0.8.7

 

March 2005

Adrian Schuur  schuur@de.ibm.com

 

 

sfcb is now an open source project - it is hosted as a SBLIM sub-project on http://sblim.sourceforge.net.

This package contains programs and libraries comprising SFCB base functions. The main program, sfcBroker is a high-performance CIM Object Manager server supporting an initial set of CIM operations, requiring very little resources.

 

Operations supported are: GetClass, EnumerateClasses, EnumerateClassNames, GetInstance, DeleteInstance, CreateInstance, ModifyInstance, EnumerateInstances, EnumerateInstanceNames, Associators, AssociatorNames, References, ReferenceNames, ExcQuery and InvokeMethod.

 

These functions are accessible via a CIM Client using the CIM XML over the HTTP/HTTPS protocol. Basic authentication is supported as an option.

 

sfcBroker will work with CMPI style providers only, the cmpi-base provider package of the SBLIM project will work, in fact, most testing has been done with these providers.

4.3         Debugging Providers

sfcBroker can be halted prior to invoking the first function request of a provider after the provider has been loaded.

To do this, the environment variable SFCB_PAUSE_PROVIDER has to be exported with the provider name. When sfcBroker is at the point of invoking the provider it will go into a loop displaying every 5 seconds the process id of the provider process in providerDrv.c. At this point a debugger can be used to attach the provider process.

From here on, debug commands must be used to get out of the sleep(5) loop.

            1: set a breakpoint at the sleep(5) statement 1817 and continue

            2: set next instruction to the first statement after the sleep loop (1823)

            3: set any other breakpoints you need and continue.

 

The following is a snapshot of the relevantcode area in providerDrv.c:

 

1817   if (once && debugMode && parms->req->operation != OPS_LoadProvider) for (;;) {

1818      fprintf(stdout,"-#- Pausing for provider: %s -pid: %d\n",name,currentProc);

1819      once=0;     

1820      sleep(5);

1821   }

1822

1823   if (parms->req->operation == OPS_LoadProvider || debugMode) {

1824      processProviderInvocationRequestsThread(parms);

1825   }

1826   else {

1827      pthread_create(&t, &tattr, (void *(*)(void *))

1828          processProviderInvocationRequestsThread, (void *) parms);

1829   }

 

A similar concept is used for pausing and debugging the HTTP Daemon. Use export SFCB_PAUSE_CODEC=http prior to starting sfcBroker and follow the steps outlined or pausing providers.

5.1         ProviderRegister

This text format control file contains provider registration data needed by sfcBroker to load and use providers. The stanza format is as follows:

 

[<class-name>]

   provider: <provide-name>

   location: <library-name>

   type:            [instance] [association] [method] [indication]                            [property]

   group:           <group-name>

   unload:         never

   namespace: fully-qualified-namespace [...]

Whereby         

<class-name>

Required

the CIM class name

<provider-name>

Required

the CMPI provider name as defined by CMPI

<location-name>

Required

the name of the provider module

type:

Required

the provider type, can be any combination of instance, association, method, indication, property

<group-name>

Optional

Enables grouping of providers in a single process. All providers using the same <group-name> will execute in the same provider process.

unload:

Optional

Defines the unload policy for this provider. The only option supported at the moment is never, meaning, the provider will not be subjected to idle time monitoring.

namespace:

Required

one or more namespace specification in which this provider can execute, for instance: root/cimv2

 

Example:

[Linux_RunningOS]

   provider: OSBase_RunningOSProvider

   location: cmpiOSBase_RunningOSProvider

   type:     instance association      

   group:    stableProvides

   namespace: root/cimv2

[Linux_OSProcess]

   provider: OSBase_OSProcessProvider

   location: cmpiOSBase_OSProcessProvider

   type:     instance association

   group:    stableProvides

   namespace:       root/cimv2   

5.2         classSchemas

This file contains all class definitions a particular instance of sfcBroker can work with. This file is created by the mof compiler mofc. Mofc executes against mof files, it requires all ancestor classes to be available when compiling. The best way to ensure this is to download a complete set of CIM mof files from the DMTF web site. For your convenience a dmtf subdirectory is included containing the CIM version 2.8 schema definitions.

A README file for mofc can be found in the ./sfcb-cvs/mofc directory.

Use sfcb/home/genMofs.sh as an example, it generates classSchemas files for the InterOp provider and the SBLIM cmpi-base package.

Every regression test must have schema data; it will be used by build.sh to build a classSchemas file.

Any *.mof file found in the ./MyTest/schema directory will be incorporated into the classSchemas file. There must be at least one .mof file.

6.2.2                        Registration data

Every regression test must have registration data; it will be collected by build.sh and appended to the providerRegister file.

Any *.reg file found in the ./MyTest/schema directory will be used. There must be at least one .reg file.

6.2.3                        Test scripts

Test scripts must be placed in sfcb/test/MyTest/tests. Any *.sh file found in this directory will be invoked by test.sh. Test scripts should return 0 when successful or 1 when not. Test scripts can/must do whatever is needed to do the test and should generate diagnostic messages when needed.

6.2.4                        Supporting scripts

Two scripts are made available to make test environments easier.

wbemcat is a simple Perl script that interfaces to the nc command. It accepts a DMTF conforming CIM-XML operation request. This request is augmented with a minimal HTTP header and sent to a CIMOM. It will wait for a response and write the response unaltered to stdout.

catdiff is a shell script that works together with wbemcat. It is used to send a .xml file containing a DMTF conforming CIM-XML operation request to a CIMOM using wbemcat and to compare its response with a file containing the expected response. When no response file is found (designated by the .OK file type) the response is stored as the .OK file and is assumed to be valid for later reruns of this test. 

Both programs are can be found in the sfcb/test directory. The scripts are used in test/Indications are for testing indication support.

8.1         Namespace support

SFCB Beta 3-0.8.6 now has full namespace support. For the moment, namespaces must be defined using the mkdir command.

Namespaces live within the $SFCB_HOME/repository directory and are themselves materialized as sub-directories of $SFCB_HOME/repository.  If  $SFCB_HOME is not defined, ./repository will be assumed.

The full path of a namepsace is $SFCB_HOME/repository/<namespace-name>. Examples of namespace-names are

            root

            root/cimv2

            root/interop

the respective full path names are

     $SFCB_HOME/repository/root

     $SFCB_HOME/repository/root/cimv2

     $SFCB_HOME/repository/root/interop

Namespace-names MUST be lower-case. The ClassProvider will translate any namespace specification into lower-case format before attempting to locate the object.

At startup time of the ClassProvider, namespaces will be discovered by traversing the $SFCB_HOME/repository directory contents. In order to be useful, namespaces must contain a classSchemas file. home/genMofs.sh shows how these files are generated for the $SFCB_HOME directory.

10          SFCB Concepts

A major SFCB objective is to provide a CIM infrastructure that is inherently stable, in the sense that it can recover from errors in providers and communication adapters. It does this primarily by running providers and adapters in an out-of-process environment. The providers and adapters may falter; the broker as such remains functional.

sfcBroker itself is the main process, it is only responsible for starting adapters, starting providers on request and determining which providers should be called for a given CIM operation. All other CIMOM functions are (will be) delegated to providers. One provider is always present: the class provider. 

Starting adapters and providers is done using the fork() system call. Communication between the three entities (broker, providers and adapters) is done using Unix Domain socket pairs. These pairs are always connected to each other, whenever a message is transmitted to one of the entities it will be accompanied with a return socket.

Currently, sfcBroker forks at most two adapter daemons, the HTTP and HTTPS daemon as specified in the sfcb.cfg file. Role of the daemon is to monitor communication ports, in case a requests arrives, an adapter codec is started again using fork(). The codec will do the actual decoding of the request driving the providers and encoding of the response.

Once the request is completely handled, the codec process exits and disappears. This may seem excessive in terms of overhead; actually it is not, performance measurements shows SCFB to be more responsive than other leading CIMOMs.  The real advantage however is that no long term effects, like storage leaks will accumulate over time.

An adapter codec asks sfcBroker which providers should be used for a particular request. As a response, the codec will get one or more sockets that it should use to drive the required providers to satisfy the request.

The codec interfaces to sfcBroker by using the getProviderContext() service routine in providerMgr.c. This routine adds provider information to the BinRequestContext structure. It then uses either the invokeProvider() or invokeProviders() routine to interface to  the provider(s) needed for this request.

 

Providers processes are started by sfcBroker as a side effect of a valid getProviderContext() call (using fork()) when not yet started because of prior requests. Provider will be unloaded after a certain amount of idle time; however, this is controlled by the sfcBroker main process. Instead, it is done autonomously by the provider process support.

The status and availability of providers is maintained automatically with System V semaphores using the SEM_UNDO option where appropriate without any communication overhead between the sfcBroker main process and the providers. A similar scheme is used to automatically throttle incoming CIM Client requests.

A socket pair will be constructed just prior to fork(), one end of this connection is passed to getProviderContext() caller, the other end is used by the provider process  itself to wait for requests.

The first request passed to the provider process is to load and initialize a specific CMPI style provider. Once loaded, every request is executed as a thread, supporting multiple provider requests in parallel.

When provider grouping is used, multiple providers will be leaded in the same provider process. 

10.1   Extension Provider

In addition to normal providers, sfcb uses Extension Provider to extend its functionality. Extension Providers are like normal providers, however not specific to a specific CIM schema. Instead, they provide specific CIMOM functionality that can be designed such that it suites specific embedded environment requirements.

Extension Providers often are method providers as well, this gives us more flexibility. Three Extension Providers are defined, they are registered as follows:

[$ClassProvider$]

   provider: ClassProvider

   location: sfcClassProvider

   type: instance method

   unload: never

#

[$DefaultProvider$]

   provider: InternalProvider

   location: sfcInternalProvider

   type: instance association method

#

[$InterOpProvider$]

   provider: InteropProvider

   location: sfcInteropProvider

   type: instance association method

#

10.1.1                   $ClassProvider$

The class provider is responsible for all schema related operations. It enables access to CIM classes and also maintains the inheritance tree. Although not currently implemented, the class provider will control which schema related operations are supported.

The getClass, enumerateClasses and enumerateClassNames requests are passed as getInstance, enumerateInstances and enumerateInstanceNames requests. Special support is included to transparently ship classes as instances between the class provider and the caller.

Other schema operations will be implemented as method requests. It then is up to the class provider to support these operations or respond with NOT_SUPPORTED.

For specific embedded environments, the class provider could keep the classes it supports as binary images that are part of the provider itself. The provider supported by Beta 3-0.8.6 uses the classSchemas file as generated by mofc.

10.1.2                   $DefaultProvider$

The default provider kicks in when no provider is registered for a specific class. It acts as an instance and association provider. It also functions as a method provider but only to respond with NOT_SUPPORTED. The default provider can maintain a repository (as does the Beta 3-0.8.6 version), but can also be implemented such that it blocks all operations with NOT_SUPPORTED.

The current implementation uses files to store instances. More advanced implementations could map repository operations to more advanced database implementations.

10.1.3                   $InterOpProvider$

The InterOp provider maintains classes as defined by the DMTF InterOp workgroup. Or, to be more precise, it is the place where InterOp specifications can be implemented. Currently it supports two classes CIM_IndicationFilter and CIM_IndicationSubscription.

Any operation against the designated InterOp namespace will be routed to the InterOp provider. Its implementation will determine to what degree the InterOp specifications are supported.

For indication support a special method (_deliver) is available to organize the delivery of indications. The CBDeliverIndication CMPI macro call uses this method to interface with the indication delivery mechanism.

Notice that indication handlers are not part of the InterOp provider. They are registered and implemented as providers and interface with the InterOp provider using the _addHandler and _removeHandler methods. Indication handlers themselves must support the _deliver method. This will be invoked by the InterOp provider when it determines that an indication has to be delivered.

Notice that the methods used and offered by the InterOp provider and indication handler are not reflected in MOF files; they are assumed to be there. Because they start with an underscore character they cannot be used by CIM Clients.

Notice also that the current implantation depends heavily on the default provider for persistency.

10.2   Indication handler

CIM_ListenerDestinationCIMXML is supported as indication handler. Its implementation is found in indCIMXMLHandler.c and indCIMXMLExport.c. It uses the curl library to generate DMTF conforming export requests.

[CIM_ListenerDestinationCIMXML]

   provider:  IndCIMXMHandler

   location:  sfcIndCIMXMHandler

   type:      instance association method

   namespace: root/interop

 

The destination property is used to determine the receiving location. When the destination starts with the file:// prefix, the remainder of the string is considered to be a local file name. The indication will then be appended to this file.

This handler responds to the _deliver method and uses the InterOp provider methods _addHandler and _removeHandler to coordinate creation and deletion of indication handler instances.
